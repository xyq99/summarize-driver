在 RK3568 这种级别的 SoC 中，**eMMC** 确实是最主流的程序存储介质，但它并非唯一选择。

RK3568 设计了非常灵活的启动架构，支持多种存储介质。你可以把它理解为一个“多面手”，根据产品成本和性能需求，程序可以存放在以下几种地方：

### 1. 核心存储：eMMC (最常用)

这是大多数 RK3568 开发板（如 Firefly、迅为、正点原子等）的首选。

- **特点：** 性能强（5.1标准）、容量大（通常 8GB~128GB）、焊接在 PCB 上非常稳定。

- **存放内容：** 整个系统的所有组件——从 Bootloader、Kernel 到庞大的 Buildroot 或 Ubuntu 文件系统，全部塞在这里。

### 2. 外部辅助：SD 卡 (MicroSD)

- **特点：** 方便插拔，主要用于**烧写固件**（用 SD 卡启动后把程序刷进 eMMC）或作为临时扩展。

- **启动顺序：** RK3568 内部的 ROM Code 通常会先检查 SD 卡，如果 SD 卡里有合法的启动程序，它会**优先从 SD 卡启动**。

### 3. 低成本方案：SPI Flash (NOR/NAND)

- **特点：** 容量小（通常 8MB~256MB），价格极低。

- **应用：** 如果你的 Buildroot 文件系统裁剪得极小，或者只需要运行一个简单的 RTOS，可以只用一颗 SPI Flash。

- **组合拳：** 有些设计会用 **SPI Flash 存 U-Boot** + **SATA 硬盘/NVMe SSD 存文件系统**。

### 4. 工业级/高性能：SATA 3.0 或 NVMe SSD

- **特点：** RK3568 拥有原生的 PCIe 3.0 和 SATA 接口。

- **模式：** 虽然 ROM Code 无法直接从 PCIe NVMe 启动，但可以先从 eMMC/SPI Flash 启动 **U-Boot**，再由 U-Boot 去挂载 **SSD** 上的内核和文件系统。这在 NAS 或边缘计算网关中很常见。

### RK3568 启动时的“搜索路径”

当电源开启后，RK3568 芯片里那段写死的 **ROM Code** 会像“侦察兵”一样按顺序扫描以下接口，寻找启动信号（Loader）：

1.  **SPI Flash** (FSPI 接口)

2.  **eMMC** (SDMMC8 接口)

3.  **SD 卡** (SDMMC0 接口)

4.  **USB OTG** (如果以上都没找到，进入从机烧写模式，等待电脑刷机)

### 总结

对于你提到的 RK3568 项目：

- **物理载体：** 绝大多数情况下程序是存在 **eMMC** 里的。

- **逻辑区分：** 它是\*\*“NAND 存储架构”\*\*（eMMC 本质也是 NAND），不支持像你之前问的 NOR Flash 那样“就地执行 (XIP)”。

**补充一点：** 由于你之前提到正在做 **TI138** 这种内存受限平台的移植，换到 **RK3568** 后你会发现“阔绰”了很多。在 RK3568 上，你几乎不需要担心程序太大跑不动，更多的是在考虑如何利用它的 **NPU** 算力和**多媒体编解码**能力。
