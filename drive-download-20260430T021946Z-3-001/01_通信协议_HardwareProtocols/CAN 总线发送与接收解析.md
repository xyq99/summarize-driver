在 CAN 总线通信中，\*\*发送单元（Transmitting Unit）**和**接收单元（Receiving Unit）\*\*并不是两个独立的物理设备，而是每个 CAN 节点（Node）都具备的功能模块。

根据 CAN 协议（ISO 11898），一个标准的 CAN 节点通常由 **CAN 控制器**和 **CAN 收发器**组成。以下是它们在发送和接收过程中的角色分配：

### 1. 发送单元 (Transmitting Unit)

发送单元的职责是将内核（如 RK3568 的 CPU）想要发送的数据转换成物理信号，并赢得总线的使用权。

- **CAN 控制器 (Controller)：**

  - **组帧：** 将原始数据包装成 CAN 报文帧（加入 ID、IDE、DLC、CRC 等字段）。

  - **仲裁：** 检查总线是否空闲，并在发送 ID 时监听回读信号。如果赢得仲裁，则继续发送。

  - **计算 CRC：** 自动生成循环冗余校验码。

- **CAN 收发器 (Transceiver)：**

  - **信号转换：** 将控制器输出的逻辑电平（TTL/CMOS）转换为差分电平（CAN_H, CAN_L）。

  - **驱动总线：** 产生足够的电流来改变总线上的电压。

### 2. 接收单元 (Receiving Unit)

接收单元的职责是监听总线上的信号，过滤出自己关心的报文，并反馈确认信号。

- **CAN 收发器 (Transceiver)：**

  - **差分转换：** 将总线上的差分电平（CAN_H - CAN_L）转换回逻辑电平发送给控制器。

- **CAN 控制器 (Controller)：**

  - **过滤器 (Acceptance Filter)：** 根据预设的规则（如屏蔽位和标识符），决定是否接收该报文。如果不符合规则，报文会被硬件直接丢弃，不干扰 CPU。

  - **校验与应答 (ACK)：** 检查 CRC 是否正确。如果正确，在 **ACK Slot** 位发送一个显性位，向发送方确认“我收到了”。

  - **解包：** 将数据段提取出来，存入 Rx 缓冲区，并触发中断通知 CPU 提取。

### 3. 两者的动态关系

在一次完整的通信过程中，发送和接收单元是高度协同的：

| **阶段**   | **发送单元的操作**             | **接收单元的操作**                     |
|------------|--------------------------------|----------------------------------------|
| **数据段** | 逐位发送数据。                 | 逐位采样数据，计算本地 CRC。           |
| **CRC 段** | 发送计算出的校验码。           | 将本地计算的 CRC 与接收到的 CRC 比较。 |
| **ACK 段** | **发送隐性位**并监听回读。     | 如果校验通过，**强行发出显性位**。     |
| **判定**   | 若监听到显性位，认为发送成功。 | 将数据存入 Buffer，通知系统。          |

### 4. 工程师视角的注意事项

对于你正在开发的 **RK3568** 平台：

- **发送队列 (Tx Queue)：** RK3568 的 CAN 控制器通常有多个发送缓冲区。如果总线负载高，发送单元会由于不断仲裁失败导致报文在队列中堆积，甚至产生 Buffer Overflow。

- **过滤规则 (Hardware Filtering)：** 在 Linux 驱动中合理配置接收单元的过滤器，可以极大地减轻 CPU 的负担。不要让所有的总线报文都产生中断，只接收该节点需要的 ID。
