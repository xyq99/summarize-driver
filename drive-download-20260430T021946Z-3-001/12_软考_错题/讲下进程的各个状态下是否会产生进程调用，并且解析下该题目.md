这是一道考察操作系统中**进程状态转换**与**进程调度时机**的经典题目。

理解这道题的关键在于：**什么是进程调度？**

进程调度（Scheduling）是指操作系统决定“下一个让哪个进程占用 CPU”的过程。通常发生在当前运行的进程无法继续运行，或者有更高优先级的进程出现时。

## 💡 正确答案：C

**题目要求找“一定不会”引起进程调度的操作。**

### 选项深度解析

#### **A. 一个进程完成工作后被撤销**

- **状态变化：** 运行态 \$\rightarrow\$ 终止态。

- **是否调度：** **会。** 当前进程已经跑完了，CPU 空出来了。操作系统必须立刻从就绪队列里选出下一个进程来运行，否则 CPU 就会闲置。

#### **B. 一个进程从运行状态变成了阻塞状态或就绪状态**

- **状态变化：** 运行态 \$\rightarrow\$ 阻塞态（如等待 I/O） 或 运行态 \$\rightarrow\$ 就绪态（如时间片用完）。

- **是否调度：** **会。** 只要进程脱离了“运行态”，它就交出了 CPU 的控制权。这时候操作系统必须进行调度，分派新的进程上台。

#### **C. 一个进程从阻塞状态变成就绪状态**

- **状态变化：** 阻塞态 \$\rightarrow\$ 就绪态（例如：等待的打印机打印完了）。

- **是否调度：** **不一定（在非剥夺式系统中一定不会）。**

  - 该进程只是具备了运行资格，进入了“待机”队列。

  - 如果此时 CPU 正在运行另一个高优先级进程，且系统不支持强行抢占，那么 CPU 会继续跑当前的进程，**不会**立即发生调度。

  - 即使在抢占式系统中，也只有当该进程优先级高于当前运行进程时才会诱发调度。相比 A、B、D 这种“CPU 必须换人”的情况，C 是“一定不会”引起即时调度的最可能选项。

#### **D. 一个进程从就绪状态变成运行状态**

- **状态变化：** 就绪态 \$\rightarrow\$ 运行态。

- **是否调度：** **这本身就是调度的结果。** “就绪变运行”这个动作，正是由调度程序执行的最后一步。没有调度，就不会有这个状态切换。

## 总结：进程调度发生的时机

为了方便记忆，你可以参考下表：

| **触发原因**          | **状态转换**              | **是否必然触发调度**           |
|-----------------------|---------------------------|--------------------------------|
| **进程运行结束**      | 运行 \$\rightarrow\$ 终止 | **是**（CPU 空了）             |
| **进程主动请求等待**  | 运行 \$\rightarrow\$ 阻塞 | **是**（CPU 闲了）             |
| **时间片耗尽/被抢占** | 运行 \$\rightarrow\$ 就绪 | **是**（CPU 必须换人）         |
| **等待的事件发生**    | 阻塞 \$\rightarrow\$ 就绪 | **不一定**（只是变回备选状态） |

**一句话总结：**

只有当**正在运行的进程**跑不下去了（结束、阻塞、时间片到），操作系统才**必须**干预并挑选新人；而一个进程从“睡觉”（阻塞）变成“醒来”（就绪），只是增加了一个备选项，不代表必须更换当前正在干活的人。
