简单直接地回答你：**reg 属性描述的是物理地址（Physical Address）或单元地址（Unit Address），绝对不是虚拟地址（Virtual Address）。**

在嵌入式开发和内核驱动领域，理解这一点至关重要。以下是详细的解析：

### 1. 为什么是物理地址？

设备树（Device Tree）的本质是**硬件的静态描述**。它告诉内核：硬件设备在 SoC 的总线拓扑结构中处于什么位置。

- **物理地址**是硬件层面的真实存在（比如 CPU 访问内存总线时的地址）。

- **虚拟地址**是内核启动后，通过 MMU（内存管理单元）建立的映射。设备树在编写时，内核甚至还没运行，因此它无法也不应该包含虚拟地址。

### 2. reg 属性的构成

reg 通常成对出现，格式为 \<address length\>。

- **Address**: 设备寄存器块的起始**物理地址**。

- **Length**: 寄存器块的大小。

例如：

uart0: serial@101f1000 {  
reg = \<0x101f1000 0x1000\>;  
};

这里的 0x101f1000 就是 UART 控制器在芯片手册（Datasheet）中标注的物理基地址。

### 3. 单元地址 (Unit Address)

你提到的“单元地址”其实是指节点名称中 @ 后面的部分（如上面例子中的 101f1000）。

- **规范要求**：如果节点有 reg 属性，那么节点的单元地址**必须**与 reg 属性中的第一个地址保持一致。

- 它的作用主要是为了在同一层级下区分同类设备（例如有多个 UART）。

### 4. 物理地址如何变成虚拟地址？

既然 reg 里是物理地址，那么你在编写 Linux 驱动程序时，不能直接解引用这个地址（否则会触发 Segmentation fault 或 Kernel Panic）。

在驱动中，你需要通过以下流程完成转换：

1.  **内核解析**：内核启动时会解析 DTS，将 reg 里的物理信息提取出来。

2.  **映射转换**：驱动程序调用 ioremap() 或 devm_ioremap_resource() 函数。

3.  **结果**：MMU 会为该物理地址建立一个页表映射，返回一个**虚拟地址**（Virtual Address）。

4.  **访问**：驱动使用 readl() / writel() 等函数操作这个返回的虚拟地址，从而间接控制硬件。

### 总结

| 维度         | 设备树中的 reg         | 驱动程序中的变量     |
|--------------|------------------------|----------------------|
| **地址类型** | **物理地址**           | **虚拟地址**         |
| **确定时间** | 硬件设计/固件编写阶段  | 内核运行映射阶段     |
| **是否唯一** | 是（对应硬件总线位置） | 否（由内核动态分配） |

所以，当你查阅 SoC 手册看到的那个寄存器基准地址，就是你应该写在 reg 里的值。
