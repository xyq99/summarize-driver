简单来说，**USB Host（USB 主机）** 就是 USB 连接关系中的“大脑”或“管理者”。

在 USB 的世界里，身份是非常明确的：必须有一个“主人”（Host）来管理一群“从设备”（Devices/Peripherals）。

### \## 1. 它扮演的角色：管理者

USB 协议采用的是典型的 **主从架构（Master-Slave）**。USB Host 负责掌控一切：

- **分配地址：** 当你插上一个 U 盘时，Host 会给它分配一个唯一的地址。

- **枚举设备：** Host 会询问设备：“你是谁？你能干什么？你需要多大的带宽？”。

- **发起通信：** 在 USB 2.0 及以前，所有的通信必须由 Host 发起。没有 Host 的指令，从设备（如鼠标）是不准主动开口说话的。

- **供电管理：** Host 负责向从设备提供 5V 电源。

### \## 2. 常见的 USB Host 硬件

- **电脑（PC）：** 最标准的 Host。主板上集成了 **USB 控制器**。

- **手机：** 现代手机通过 **OTG（On-The-Go）** 技术，可以在从设备（连接电脑传文件时）和主机（连接键盘、U 盘时）身份之间切换。

- **嵌入式板卡（如你的 RK3568）：** 这种 SoC 内部通常集成了多个 USB 控制器，有的专门跑 Host 模式（接外部模块），有的支持 OTG。

### \## 3. USB Host 与 Device 的区别

| **特征**     | **USB Host (主机)**              | **USB Device (从机/外设)**               |
|--------------|----------------------------------|------------------------------------------|
| **典型例子** | 电脑、笔记本、手机 (OTG)         | U 盘、键盘、摄像头、打印机               |
| **控制权**   | 拥有绝对控制权，发起所有传输     | 响应主机的请求，被动传输                 |
| **物理接口** | 通常是 Type-A 母口或 Type-C      | 通常是 Type-B、Micro-USB 或线缆末端      |
| **驱动需求** | 需要安装复杂的通用主机控制器驱动 | 需要具体的类驱动（如 HID、Mass Storage） |

### \## 4. 核心组件：HCD (Host Controller Driver)

作为一名嵌入式工程师，你可能更关心底层实现。一个完整的 USB Host 系统包含三层：

1.  **USB 硬件控制器：** 如 EHCI (USB 2.0)、XHCI (USB 3.0)。

2.  **HCD (主机控制器驱动)：** Linux 内核中负责与硬件控制器打交道的底层软件。

3.  **USB 核心层 (USB Core)：** 处理协议逻辑（如地址分配、配置解析）。

### \## 5. 什么是 USB OTG？

既然有 Host 和 Device 之分，如果两个设备都想当主人（或者都只能当从机）怎么办？

**USB OTG (On-The-Go)** 解决了这个问题。它允许设备通过检测接口引脚（ID 脚）的状态，动态地切换自己的身份：

- 如果 ID 脚接地，设备就变成 **Host**。

- 如果 ID 脚悬空，设备就变成 **Device**。

**通俗类比：**

USB Host 就像是**交警**，USB Device 就像是**过往车辆**。没有交警指挥，车辆不知道什么时候该走，也不知道该走哪条道；而交警手上握着电源开关和通行证发号器。

你在做 RK3568 开发时，如果想外接一个 USB 摄像头，你就得确保你的内核驱动里 **USB Host Controller** 已经正确加载，并且设备树（DTS）里的模式配置为了 host 或 otg。
