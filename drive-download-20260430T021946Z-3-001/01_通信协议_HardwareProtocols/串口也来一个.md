在 RK3568 上，串口（UART）的驱动层级与 I2C/SPI 有很大不同。由于串口是 Linux 极其核心的设备（涉及控制台终端），开发者通常**不需要**像 I2C/SPI 那样编写一个从零开始的 uart_driver。

相反，RK3568 的 UART 驱动已经由 Rockchip 官方在内核中写好了（基于 8250 或 dw-apb-uart）。你通常只需要在 **DTS** 中开启它，然后在代码中直接调用 **TTY 核心层** 的接口。

## 1. 设备树描述 (DTS)

假设你使用 RK3568 的 uart3 连接一个外设（如 GPS、传感器或蓝牙），且不需要硬件流控。

&uart3 {  
status = "okay";  
pinctrl-names = "default";  
pinctrl-0 = \<&uart3m0_xfer\>; // 配置 TX 和 RX 引脚  
  
/\* 如果外设需要特定的驱动控制（例如蓝牙芯片），可以在此添加子节点 \*/  
/\* 如果只是通用串口通讯，通常保持空节点即可 \*/  
};

## 2. 内核空间接口 (Kernel Space)

在内核驱动代码中，如果你需要读取串口数据，通常有两条路：

### 方法 A：通过 TTY 核心层（推荐用于外设驱动）

如果你的驱动是挂在 UART 下的子驱动（例如使用 serdev 框架），代码如下：

\#include \<linux/module.h\>  
\#include \<linux/serdev.h\>  
  
struct my_uart_dev {  
struct serdev_device \*serdev;  
};  
  
/\* 1. 接收回调：当串口收到数据时内核自动调用 \*/  
static int my_uart_receive_buf(struct serdev_device \*serdev, const unsigned char \*buf, size_t count)  
{  
// 处理接收到的数据 buf  
pr_info("Received %zu bytes from UART3\n", count);  
return count;  
}  
  
/\* 2. 核心控制函数 \*/  
static const struct serdev_device_ops my_serdev_ops = {  
.receive_buf = my_uart_receive_buf,  
};  
  
static int my_uart_probe(struct serdev_device \*serdev)  
{  
struct my_uart_dev \*mdev;  
  
mdev = devm_kzalloc(&serdev-\>dev, sizeof(\*mdev), GFP_KERNEL);  
if (!mdev) return -ENOMEM;  
  
mdev-\>serdev = serdev;  
serdev_device_set_client_ops(serdev, &my_serdev_ops);  
  
/\* 打开串口并设置波特率 \*/  
serdev_device_open(serdev);  
serdev_device_set_baudrate(serdev, 115200);  
serdev_device_set_flow_control(serdev, false);  
  
/\* 发送数据示例 \*/  
serdev_device_write(serdev, "Hello RK3568 UART", 18, 100);  
  
return 0;  
}  
  
static const struct of_device_id my_uart_of_match\[\] = {  
{ .compatible = "vendor,uart_sensor", },  
{ /\* sentinel \*/ }  
};  
  
static struct serdev_device_driver my_uart_driver = {  
.driver = {  
.name = "my_uart_driver",  
.of_match_table = my_uart_of_match,  
},  
.probe = my_uart_probe,  
};  
  
module_serdev_device_driver(my_uart_driver);

## 3. 驱动层关键接口对比

| 接口类型       | 接口函数                     | 特点                                          |
|----------------|------------------------------|-----------------------------------------------|
| **发送数据**   | serdev_device_write()        | 将数据写入发送缓冲区，并支持超时设置。        |
| **接收数据**   | receive_buf 回调             | **异步通知**。不需要轮询，效率最高。          |
| **波特率设置** | serdev_device_set_baudrate() | 动态修改通信速率（如从 9600 切换到 115200）。 |
| **控制线路**   | serdev_device_set_tiocm()    | 手动控制 RTS/DTR 等引脚电平。                 |

## 4. RK3568 特别说明

1.  **调试控制台 (FIQ Debugger)**：RK3568 的 uart2 通常被内核占用作为调试控制台。如果你想把 uart2 当普通串口用，必须在 DTS 中关闭 fiq_debugger。

2.  **DMA 模式**：对于高速串口通讯（如 3Mbps 以上），建议在 DTS 中开启 dma-names = "tx", "rx";，这样可以极大降低读取数据时的 CPU 负载。

3.  **用户态测试**：

    - 串口开启后会生成 /dev/ttyS3（对应 uart3）。

    - 可以直接使用命令测试：stty -F /dev/ttyS3 115200 && echo "test" \> /dev/ttyS3。

对于你正在处理的 128 路采样数据项目，如果这些 ADC 的控制是通过串口指令下发的，使用 serdev 框架是维护性最好的方案，因为它能像 I2C/SPI 一样通过设备树管理从机。
