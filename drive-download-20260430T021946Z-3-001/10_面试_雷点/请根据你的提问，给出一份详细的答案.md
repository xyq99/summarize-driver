### 1. 异构多核通信中的数据一致性保障

在 OMAP-L138 或 Zynq-7100 这种架构中，ARM 核开启了 MMU 和 L1/L2 Cache，而 DSP 或 FPGA 侧通常通过总线直接访问 DDR（类似 DMA 行为），这必然会导致 **Cache 不一致性**。

#### 核心解决方案：

- **软件管理一致性（Cache Maintenance）：**

  - **ARM 发送数据给 DSP：** ARM 写入数据后，必须执行 **Clean (Flush)** 操作，将 Cache 中的脏数据强制推回到 DDR，确保 DSP 读取的是最新值。

  - **ARM 接收来自 DSP 的数据：** 在读取前，必须执行 **Invalidate** 操作，使当前 Cache 行失效，强制从 DDR 重新加载 DSP 写入的数据。

- **硬件管理方案（Zynq ACP 接口）：**

  - 在 Zynq 平台上，如果使用 **ACP (Accelerator Coherency Port)** 接口，FPGA 的访问可以经过 ARM 的 L2 Cache 监听控制单元（SCU），从而实现硬件级的自动一致性，无需软件手动干预 Cache。

- **性能优化策略：**

  - 对于 **1000Hz** 的高频交互，频繁的 Cache 操作开销巨大。我通常采用 **dma_alloc_coherent** 申请一段“不可缓冲（Non-cacheable）”的内存空间作为 Ring Buffer。虽然 CPU 访问稍慢，但省去了昂贵的 Cache 同步时间，总体吞吐量更高。

### 2. 16 MB NorFlash 与受限内存的极致裁剪

针对 OMAP-L138 的 16 MB NorFlash 和 Zynq-7100 的 256 MB DDR 环境，裁剪策略需要“精准手术”。

#### 裁剪实战：

- **U-Boot 深度瘦身：**

  - 禁用所有非核心命令（如 usb, net, nfs），仅保留 bootm 和关键 I/O 初始化。

  - 启用 CONFIG_TINY_PRINTF 减小二进制体积。

- **内核（Kernel）裁剪：**

  - **模块剔除：** 禁用 CONFIG_MODULES，将所有驱动静态编译入内核。剔除冗余文件系统支持（如 Ext4/NTFS），仅保留 **SquashFS**（高压缩只读）用于根文件系统。

  - **移除调试符号：** 关闭 CONFIG_DEBUG_INFO 和内核抢占追踪，显著降低 vmlinux 大小。

- **内存分配规划：**

  - 在 Device Tree 中通过 **reserved-memory** 节点手动规划内存布局。例如，将高端 64 MB 内存完全从 Linux 管理中剥离，专门留给 DSP 运行算法镜像和 60 MB 的原始数据缓冲区，防止内核 OOM（内存溢出）影响核心算法。

### 3. LwIP 协议栈在高频（1000Hz）场景下的优化

在 1000Hz 数据流下，传统的“拷贝式”处理会导致系统死机或大量丢包。

#### 优化路径：

- **零拷贝（Zero-copy）接收：**

  - 修改 MAC 驱动，使 DMA 直接将数据包存入 LwIP 的 **pbuf** 中。应用层通过 pbuf-\>payload 指针直接读取数据，避免从内核空间到应用空间的物理拷贝。

- **内存池（PBUF Pool）调优：**

  - 增大 PBUF_POOL_SIZE 以应对瞬时突发流量。

  - 针对 1000Hz 的固定长度采样包，自定义内存池 MEMP_NUM_RAW_PCB，将 pbuf 分配与算法处理周期对齐，降低内存碎片化。

- **中断下半部处理：**

  - 利用 FreeRTOS 的信号量机制，将网络包的解析从硬中断中移出，交给高优先级任务处理，确保系统不会因中断过于频繁而“饿死”低优先级任务。

### 4. Linux 5.10 新增系统调用（Syscall）的实现

在 RK3568 项目中，为了提供比 ioctl 更简洁、性能更优的算法控制接口，我选择了自定义系统调用。

#### 实现步骤：

1.  **修改系统调用表：** 在 arch/arm64/kernel/syscall.tbl 中添加新编号及函数关联（如 \_\_arm64_sys_radar_control）。

2.  **定义入口函数：** 在 kernel/sys.c 中使用 SYSCALL_DEFINE 宏定义函数。需要使用 copy_from_user 严格检查用户态传来的参数，确保内核安全。

3.  **动机解析：** 相比 ioctl 需要解析复杂的命令码，Syscall 的路径更短（直接通过软中断 svc 进入），在需要纳秒级响应的控制指令下，这种开销减量在工业实时场景中非常关键。

### 5. AI 协同开发（Vibe Coding）的质量严控

虽然 AI 能极大地提升代码编写速度，但嵌入式底层的逻辑必须经过“人类审计”。

#### 我的验证工作流：

- **提示词约束（Prompt Constraint）：** 在生成驱动代码时，我会明确要求 AI 使用 volatile 修饰寄存器指针，并强制要求生成的代码符合 **MISRA C** 安全标准。

- **逻辑对比验证：** 利用 AI 编写 Python 测试脚本（简历中提到的仿真框架），与生成的 C 代码进行“对账”测试。

- **硬件标定：** 最终逻辑必须通过简历中提到的 **逻辑分析仪** 进行时序标定。AI 可能知道 SPI 需要拉低 CS，但具体的 setup/hold time 必须根据 OMAPL138 数据手册手动在代码中微调。
