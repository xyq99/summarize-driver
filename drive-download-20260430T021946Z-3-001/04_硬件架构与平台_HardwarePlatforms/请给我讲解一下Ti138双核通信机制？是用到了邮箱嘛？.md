针对 TI C6000 系列（如经典的 OMAP-L138 或 C66x 系列）的双核通信，你的直觉非常准：**Mailbox（邮箱）确实是其最核心的硬件基础**。

但对于 TI 这样复杂的片上系统（SoC），双核通信（IPC, Inter-Processor Communication）通常不是单一的硬件读写，而是一个\*\*“硬件通知 + 共享内存”\*\*的组合拳。

## 1. 核心机制：Mailbox (硬件基础)

在 TI OMAP-L138（ARM9 + C674x DSP）中，Mailbox 模块提供了 CPU 之间互相“打招呼”的能力。

- **寄存器结构**：Mailbox 包含多组寄存器，每组包含 **Data 寄存器** 和 **Flag 寄存器**。

- **通信流程**：

  1.  **发送方（如 ARM）**：将一个 32 位的值（通常是一个指针或指令码）写入 Mailbox 寄存器。

  2.  **触发中断**：写入动作会自动触发接收方（如 DSP）的硬件中断。

  3.  **接收方（如 DSP）**：在中断服务程序（ISR）中读取该 32 位值，并清除中断标志。

**注意**：由于 Mailbox 硬件寄存器很小（只能传 32 位数据），它通常只用来传“信号”，而不直接传“大量数据”。

## 2. 协作模式：Mailbox + Shared RAM (大数据传输)

为了传输大量数据（如音视频流或采样信号），TI 采用了 **共享内存** 机制。

- **物理载体**：L138 内部有一块两核都能访问的 **Shared RAM**。

- **分工协作**：

  - **Step 1**：ARM 将 1MB 的图像数据写入 Shared RAM 的起始地址 0x80001000。

  - **Step 2**：ARM 将这个地址 0x80001000 写入 Mailbox。

  - **Step 3**：DSP 收到 Mailbox 中断，提取出地址，直接去 Shared RAM 对应的位置取数据处理。

  - **Step 4**：DSP 处理完后，再通过 Mailbox 回复 ARM 一个“处理完成”的信号。

## 3. 软件框架：IPC 与 MessageQ

如果你在开发 TI 官方的 SDK（以前叫 BIOSMCSDK 或 Processor SDK），你不会直接操作 Mailbox 寄存器，而是使用 **IPC 软件包**。

其中最常用的机制是 **MessageQ**：

- **队列化**：它在共享内存中维护了多个“消息队列”。

- **零拷贝（Zero-copy）**：MessageQ 实际上是在传递**缓存描述符**。发送方申请一块共享内存，填入数据，然后把指针发给对方。

- **通知机制**：底层依然调用的是 **Notify** 模块，而 Notify 模块的底层驱动就是你提到的 **Mailbox**。

## 4. 关键挑战：Cache 一致性

这是 TI 双核开发最容易踩的坑，比 STM32 复杂得多：

- **问题**：ARM 往共享内存写了数据，但由于 ARM 有 **Cache**，数据可能还停留在 Cache 里没进 DDR/Shared RAM。此时 DSP 去读，读到的是旧的脏数据。

- **解决**：

  - **ARM 写完后**：必须执行 Cache_wb (Write Back，回写)，强制将数据推入物理内存。

  - **DSP 读之前**：必须执行 Cache_inv (Invalidate，使无效)，强制从物理内存重新拉取数据。

## 5. 总结：TI 138 双核通信 vs STM32

| 特性           | STM32 (单核)         | TI 138 (双核)                       |
|----------------|----------------------|-------------------------------------|
| **通信介质**   | 全局变量             | **Mailbox (硬件中断) + Shared RAM** |
| \*\*通知方式   | 轮询标志位           | **硬件 Mailbox 中断**               |
| **数据一致性** | 简单（不涉及 Cache） | **非常严格（必须手动维护 Cache）**  |
| **复杂程度**   | 低                   | 高（涉及内存映射、内核隔离）        |

**一句话总结**：Mailbox 是“敲门声”，告知对方有信件；Shared RAM 是“信箱”，存放真正的信件内容。

既然你在研究 RK3568，你会发现 RK3568 的 **Mailbox** 机制与 TI 非常相似，只不过 RK3568 更多是用于 **AP (CPU) 与 MCU (RKPMU/M4)** 之间的通信。你是在尝试给 L138 编写自定义的通信协议吗？
