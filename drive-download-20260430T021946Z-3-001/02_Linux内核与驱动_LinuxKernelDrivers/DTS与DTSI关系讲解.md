在嵌入式 Linux 开发中，特别是针对像 **RK3568** 这样复杂的 SoC，设备树（Device Tree）的组织方式非常类似于面向对象编程中的“类”与“实例”的关系。

我们可以通过以下三个维度来理解 dts 和 dtsi：

## 1. 基本定义与包含关系

- **.dtsi (Device Tree Include)**：\
  可以类比为 C 语言中的 **.h 头文件**。它通常包含通用的硬件描述。例如，RK3568 的 CPU 核心数、GPU 基地址、中断控制器等所有板卡通用的部分。

- **.dts (Device Tree Source)**：\
  可以类比为 **.c 源文件**。它代表具体的“开发板”或“产品”。它会通过 /include/ 指令包含一个或多个 .dtsi 文件，并根据实际电路设计添加或修改节点。

## 2. 以 RK3568 为例的层级结构

在瑞芯微（Rockchip）的内核源码中，这种继承关系通常分为三层：

### 第一层：SoC 级 (rk3568.dtsi)

这是最基础的文件，描述了芯片内部的资源。

- **内容**：定义了 4 个 Cortex-A55 核心、内存映射、各种控制器的寄存器地址（如 I2C, SPI, UART）。

- **特点**：这些控制器的状态通常默认是 status = "disabled";，因为芯片厂商不知道你会用到哪一组。

### 第二层：核心板/通用功能级 (rk3568-evb.dtsi 等)

许多厂家会将核心板（SoC + DDR + eMMC）通用的配置抽离出来。

- **内容**：配置了电源管理 IC (PMIC) 的电压域、默认的引脚复用 (pinctrl) 等。

### 第三层：具体板级 (rk3568-pinctrl-demo.dts 或你的项目名.dts)

这是你最终编译的目标文件。

- **内容**：引用上面的文件，并根据你的电路板修改。

  - **启用外设**：比如你的项目在 UART2 上接了调试串口，你会写 &uart2 { status = "okay"; };。

  - **覆盖参数**：如果 SoC 默认 SPI 频率是 50MHz，但你的外设只支持 10MHz，你可以在这里覆盖它。

## 3. 关键机制：引用与覆盖 (Override)

这是 dts 处理 dtsi 最核心的逻辑。设备树采用**后定义覆盖先定义**的原则。

**示例代码逻辑：**

> C++

// rk3568.dtsi (芯片级)\
uart2: serial@fe660000 {\
compatible = "rockchip,rk3568-uart";\
reg = \<0x0 0xfe660000 0x0 0x100\>;\
status = "disabled"; // 默认关闭\
};\
\
// your-board.dts (你的开发板)\
/include/ "rk3568.dtsi"\
\
&uart2 {\
status = "okay"; // 覆盖上面的 disabled，开启串口\
pinctrl-0 = \<&uart2m0_xfer\>; // 指定具体的引脚复用\
};

## 总结

|                 |                     |                               |
|-----------------|---------------------|-------------------------------|
| **特性**        | **.dtsi**           | **.dts**                      |
| **角色**        | 模板、通用公共部分  | 最终产品、具体配置            |
| **包含方式**    | 被别人包含          | 包含 .dtsi 文件               |
| **编译结果**    | 不直接编译成二进制  | 编译成 .dtb 给内核读取        |
| **RK3568 对应** | rk3568.dtsi (SoC级) | rk3568-rock-3a.dts (具体板子) |

在开发 RK3568 驱动时，你绝大多数时候是在 .dts 中通过 &label 的方式修改 .dtsi 里预定义的节点，而不是去改动芯片级的 .dtsi 文件。

针对你手头那个 256MB DDR 的 Zynq 平台或者是 RK3568 开发板，如果你需要修改内存映射或开启某个特定的控制器，通常也是在最顶层的 .dts 中进行覆盖。
