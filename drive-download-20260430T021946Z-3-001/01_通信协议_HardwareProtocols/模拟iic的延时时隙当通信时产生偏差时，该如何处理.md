在嵌入式开发中，软件模拟 I2C（GPIO Bit-banging）由于不受硬件控制器限制，灵活性很高，但由于它是靠软件循环或 delay 函数控制时序，极易受到系统调度、中断或时钟频率漂移的影响。

当通信产生偏差（如数据错位、ACK 丢失或设备无响应）时，可以从以下几个维度进行处理和优化：

## 1. 严格的时序冗余设计

模拟 I2C 的核心在于确保 SDA 在 SCL 为高电平时保持稳定。如果时隙偏差导致 SCL 跳变过快，从设备可能无法采样。

- **增加延时余量**：如果 I2C 标准模式是 100kHz（周期 \$10\mu s\$），在模拟实现时，可以将半个时钟周期的延时设置得比理论值稍大（例如理论 \$5\mu s\$，实际给 \$6\sim 7\mu s\$）。

- **确保数据建立时间（Setup Time）**：在 SCL 拉高之前，先改变 SDA 并通过延时确保其稳定。

- **对称性调整**：确保 SCL 高电平和低电平的持续时间尽可能一致，避免产生过窄的脉冲导致从机硬件滤波将其滤除。

## 2. 引入中断屏蔽（关键）

在多任务系统（如 Linux 或 RTOS）中，模拟 I2C 的代码段如果被高优先级任务或硬中断打断，会导致时钟线被长时间挂起，从而产生巨大的时序偏差。

- **原子操作保护**：在发送字节或 Start/Stop 信号的紧凑时序块中，暂时关闭全局中断。  
  C  
  local_irq_save(flags); // 锁定中断  
  I2C_Send_Byte(data);  
  local_irq_restore(flags); // 恢复中断

- **禁用抢占**：在内核态下，使用 preempt_disable() 确保当前线程不被切走。

## 3. 时钟拉伸（Clock Stretching）处理

很多开发者在模拟 I2C 时会忽略从机对时钟的控制权。如果从机处理速度慢，它会拉低 SCL 线以阻止主机继续发送。

- **SCL 回读检测**：在主机释放 SCL（置高电平）后，**必须**读取 SCL 引脚的状态。如果检测到 SCL 依然是低电平，说明从机正在拉伸时钟，主机必须循环等待直到 SCL 变为高电平后，才能进行后续的延时和采样。  
  C  
  SCL_H();  
  while(READ_SCL() == 0); // 等待从机释放时钟  
  I2C_Delay(); // 此时才开始计算高电平延时

## 4. 增强采样稳定性

由于电气噪声或走线电容导致的波形畸变，单次采样 SDA 可能不准。

- **中间采样法**：采样 SDA 的最佳时机是在 SCL 高电平期间的正中间，而不是刚拉高或快拉低的时候。

- **多次采样取众数**：在 SCL 高电平期间连续读取 3 次 SDA，通过“三取二”逻辑判定最终位状态，可以有效过滤高频尖峰干扰。

## 5. 错误恢复机制（Bus Recovery）

如果偏差已经导致通信卡死（例如从机正在等待时钟而主机认为已经结束），需要有一套复位逻辑。

- **发送 9 个时钟脉冲**：当总线挂死（SDA 一直为低）时，主机可以模拟发送 9 个连续的 SCL 脉冲。这会强制让正在输出“0”的从机释放 SDA，随后再发送一个 Stop 信号重新初始化通信。

- **软件超时重试**：为每一次 while 等待（如等待 ACK）设置超时计数器，一旦超时则立即调用复位函数，避免程序进入死循环。

## 6. 物理层补偿

- **上拉电阻阻值**：如果偏差是由于波形上升沿过缓（电容效应）导致的，尝试减小上拉电阻（如从 10k 换成 2.2k/4.7k）。

- **GPIO 配置**：确保 GPIO 配置为 **开漏输出（Open-Drain）** 模式，而不是推挽模式通过切换方向来模拟。

### 总结

处理模拟 I2C 偏差的准则是：**时序宁慢勿快、采样必须在中心、务必处理时钟拉伸、关键路径屏蔽中断。**
