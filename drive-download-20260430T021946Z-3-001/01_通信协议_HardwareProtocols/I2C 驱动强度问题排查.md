既然你想看，那我们直接从**底层电物理**的角度，通过“脑补”示波器波形来看驱动强度（Drive Strength）不足对 I2C 通信的破坏。

在 RK3568 这种高性能芯片上，如果 drive-strength 设置过小（比如 2mA），而你外部接了比较长的排线或者挂了 5、6 个传感器（寄生电容大），你会看到以下三种典型的“车祸现场”：

## 1. 典型的“圆角矩形”与采样失败

正常波形应该是像方砖一样的矩形，但驱动力不足时，电荷像蜗牛爬一样充进电容里。

- **现象：** SCL 变高后，SDA 还在努力向上爬，等它爬到逻辑“1”的电压阈值（例如 \$0.7 \times V\_{CC}\$）时，SCL 的高电平周期可能已经结束了。

- **后果：** 接收方采样时，发现电压还在中间状态，于是误判为“0”。

- **Linux 报错：** 你会在 dmesg 里看到 i2c-rockchip: timeout 或 i2c transfer failed (-110)。

## 2. “伪”起始/停止信号 (False Start/Stop)

这是最诡异的错误。由于驱动力弱，信号沿变得非常缓。

- **现象：** 假设此时 SCL 已经是高电平（采样区），而 SDA 因为前一个动作的驱动力太弱，还没完全回到稳定状态，产生了一个微小的波动或缓慢的滑落。

- **后果：** 这种在 SCL 高电平期间的电平滑动，极易被 I2C 控制器误识别为 **START** 或 **STOP** 信号。

- **结果：** 通信突然中断，总线死锁（Bus Busy），甚至导致整个 I2C 控制器挂起。

## 3. 如何在 RK3568 的 DTS 中“救火”？

如果你发现通信不稳定，可以尝试修改内核源码中的设备树（Device Tree）。找到对应的 pinctrl 配置：

> DTS

/\* 示例：修改 RK3568 的 I2C2 引脚驱动能力 \*/  
&i2c2 {  
pinctrl-names = "default";  
pinctrl-0 = \<&i2c2m0_xfer\>; // 引用具体的引脚配置  
status = "okay";  
};  
  
&pinctrl {  
i2c2 {  
i2c2m0_xfer: i2c2m0-xfer {  
/\* rockchip,pins 格式通常为: \<行号 编号 功能 属性\> \*/  
rockchip,pins =  
\<0 RK_PB1 1 &pcfg_pull_up_drv_12ma\>, // 将驱动能力提到 12mA  
\<0 RK_PB2 1 &pcfg_pull_up_drv_12ma\>;  
};  
};  
};

**归纳总结：**

- **低驱动能力** = 信号沿变斜 = 建立时间（Setup Time）不足 = 采样读到错误数据。

- **高驱动能力** = 信号沿陡峭 = 容易产生过冲（Overshoot） = 可能引入高频干扰。
