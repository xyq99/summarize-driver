在 Linux 内核中，中断处理被设计为两个阶段：**上半部（Top Half）和下半部（Bottom Half）**。这种设计是为了平衡系统的实时响应能力和吞吐量。

## 1. 硬中断 vs. 软中断

### 硬中断 (Hard IRQ)

- **触发方式：** 由硬件外设（如网卡、键盘、磁盘）通过物理引脚发送电信号给中断控制器。

- **执行环境：** 运行在特定的“中断上下文”中。

- **特性：** 优先级最高，会**嵌套**或**打断**当前正在运行的进程。为了防止丢失其他硬件信号，硬中断处理函数必须**极快**。

- **限制：** 不能休眠，不能调用可能引起调度的函数。

### 软中断 (Softirq)

- **触发方式：** 由内核软件指令触发（通常是硬中断处理完后标记一个 bit 位）。

- **执行环境：** 同样运行在中断上下文中，但其执行时机通常是在硬中断退出后。

- **特性：** 允许多个 CPU 并发执行（即使是同一类型的软中断）。

- **应用：** 主要用于对性能要求极高的场景，如网络收发（NET_RX/TX）、定时器。

## 2. 中断下半部的实现方式及区别

由于硬中断必须快，复杂的逻辑会推迟到“下半部”执行。Linux 主要有三种机制：

### 软中断 (Softirq)

- **特点：** 静态编译在内核中，数量有限（目前 Linux 仅定义了 10 种）。

- **性能：** 最高，支持全对称多处理器（SMP）并行执行。

- **复杂度：** 开发最难，因为它是重入的，必须处理严格的同步问题。

### Tasklet

- **特点：** 基于软中断实现的一种封装。

- **并发性：** **非重入**。同一类型的 Tasklet 不会在两个 CPU 上同时运行。

- **使用场景：** 驱动开发者最常用的方式，简化了同步逻辑，性能虽略逊于原生的软中断，但远高于工作队列。

### 工作队列 (Workqueue)

- **特点：** 将任务交给内核线程去执行。

- **执行环境：** **进程上下文**。

- **性能：** 相对较低，涉及上下文切换的开销。

|  |  |  |  |
|----|----|----|----|
| **比较维度** | **软中断 (Softirq)** | **Tasklet** | **工作队列 (Workqueue)** |
| **上下文** | 中断上下文 | 中断上下文 | 进程上下文 |
| **串行执行** | 否（高度并发） | 是（同类不会并发） | 取决于具体配置 |
| **是否可休眠** | **不可休眠** | **不可休眠** | **可以休眠** |
| **优先级** | 极高 | 高 | 普通（跟随内核线程） |

## 3. 为什么只有工作队列能实现休眠？

这是一个内核设计架构的核心问题：

### 根源：上下文（Context）的区别

- **中断上下文（软/硬中断）：** 它们并不属于任何特定的进程。它们是“借用”了当前 CPU 上正在运行的进程的内核栈，但它们没有自己的 task_struct。如果中断处理程序进入休眠，它会尝试调用调度器（Scheduler）将自己挂起。由于没有 task_struct 来记录它的状态，调度器根本不知道以后该如何唤醒它，系统会直接崩溃（Panic）。

- **进程上下文（工作队列）：** 工作队列的任务是由专门的内核线程（如 kworker）来承载的。每一个工作项在执行时，都有一个真实的进程结构支持。

### 休眠的本质

\$\$Sleep = Yielding \\ the \\ CPU + Scheduling\$\$

当工作队列中的代码调用 msleep() 或等待信号量时：

1.  内核会将当前的 kworker 线程状态标记为“睡眠”。

2.  调度器会保存该线程的寄存器和栈信息。

3.  调度器挑选另一个就绪进程运行。

4.  当条件满足时，调度器再将 kworker 线程唤醒并恢复执行。

**总结：** 只有工作队列拥有完整的“户口”（进程描述符），所以它才能在休眠后被重新找回来继续工作；而软硬中断是“无家可归”的临时插队者，一旦休眠就会在内存中迷失。
