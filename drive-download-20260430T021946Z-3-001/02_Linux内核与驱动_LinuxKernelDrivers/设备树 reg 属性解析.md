在设备树（Device Tree）中，reg 属性的解析方式并不是固定不变的，它取决于该节点父节点的两个关键属性：**\#address-cells** 和 **\#size-cells**。

对于你提到的 RK3568 设备树片段：

reg = \<0x0 0xfe660000 0x0 0x100\>;

这里的数值含义通常是 **“起始地址”** + **“长度（大小）”**。

### 1. 为什么有 4 个值？

RK3568 是一款 64 位 SoC。在它的根节点或总线节点（如 bus@...）中，通常会定义：

- \#address-cells = \<2\>; （表示地址需要用 2 个 32 位单元来表示，即 64 位地址）

- \#size-cells = \<2\>; （表示长度也需要用 2 个 32 位单元来表示，即 64 位长度）

### 2. 详细拆解分析

按照上述 2 + 2 的逻辑，你的 reg 属性应该这样拆分：

|  |  |  |
|----|----|----|
| **数值** | **逻辑归属** | **实际含义** |
| **0x0** | **地址的高 32 位** | 起始物理地址的高位（因为 RK3568 物理空间没超过 4GB，所以高位通常为 0） |
| **0xfe660000** | **地址的低 32 位** | 起始物理地址的低位，即 **0xFE660000** |
| **0x0** | **长度的高 32 位** | 地址空间长度的高位 |
| **0x100** | **长度的低 32 位** | 地址空间长度的低位，即 **0x100** 字节 (256 字节) |

**结论：** 你的理解稍微有一点偏差。**前两个值共同组成了“起始地址”**，而**后两个值共同组成了“空间大小”**。

### 3. 特殊情况：32 位模式

如果你在一些较老的芯片（如 RK3328）或者某些特定的子总线下看到 reg = \<0xfe660000 0x100\>;，那是因为当时的：

- \#address-cells = \<1\>;

- \#size-cells = \<1\>;\
  这种情况下，第一个值就是地址，第二个值就是大小。

### 4. 为什么 RK3568 要用两个 cell 表示地址？

虽然 RK3568 的外设寄存器大多分布在 0xFE...... 这种 4GB 以内的区间，但它的内存（DDR）可以超过 4GB。为了保持寻址空间的统一性和兼容 64 位架构，规范要求使用 2 个 cells 来确保可以描述大于 0xFFFFFFFF 的物理地址。

### 工程师提示：

在调试时，如果你不确定某个驱动对应的 reg 怎么写，最稳妥的方法是向上查找它的 parent 节点。

- 如果父节点是 amba 或 bus，看它们的 \#address-cells。

- 在 RK3568 的 .dtsi 文件中，你会发现 range 属性经常配合这些 cells 进行地址转换。
