在底层开发（尤其是 ARM 或 x86 汇编）中，**函数序言（Function Prologue）** 是每个函数执行的第一段代码。它的存在是为了给函数搭建一个安全、独立的运行环境，也就是所谓的“栈帧”（Stack Frame）。

而 **SP（栈指针）递减一个偏移量**，正是为了在内存中“强行”划出一块领地，留给该函数的局部变量使用。

### 1. 什么是函数序言 (Function Prologue)？

当程序跳转到一个子函数时，CPU 就像进入了一个新房间。为了不把“前任”（调用者）的东西搞乱，并在离开时能恢复原状，子函数必须执行几行固定的汇编指令。这个过程就是函数序言。

函数序言通常完成以下三件事：

1.  **保存现场：** 将调用者的关键寄存器（如 FP, LR 或一些通用寄存器）压入栈中。

2.  **更新帧指针 (FP)：** 将当前的 SP 值赋给 FP，标志着这个新房间（新栈帧）的起点。

3.  **开辟空间：** 让 SP 向下移动（递减），为局部变量预留位置。

### 2. 为什么要让 SP 递减一个偏移量？

在大多数现代计算机架构（如 ARM, x86）中，**栈是向低地址增长的**。

- **物理意义：** 想象栈是一个倒着放的存储桶。桶底在高地址，开口在低地址。

- **递减动作：** 如果子函数定义了 int a, b;，它需要 8 个字节。编译器会自动生成一条指令，如 sub sp, sp, \#8（ARM 语法）。

- **结果：** SP 减掉 8 后，指向了更低的一个地址。从这个新 SP 到旧 SP 之间的那 8 个字节，就归这个函数独占了。

### 3. 实例解析（以 ARM 汇编为例）

假设你写了一个简单的 C 函数：

> C

void my_func() {  
int x = 10;  
int y = 20;  
}

编译器生成的**函数序言**可能长这样：

> Code snippet

push {fp, lr} @ 1. 保存前任的帧指针和返回地址到栈里 (SP 自动减 8)  
add fp, sp, \#4 @ 2. 设置当前函数的帧指针起点  
sub sp, sp, \#8 @ 3. SP 递减 8 字节，这就是为 x 和 y 开辟的“偏移量”

此时的内存布局如下：

- **FP 指向的地方：** 存着旧的 FP 和返回地址。

- **SP 指向的地方：** 到 FP 之间的这 8 字节空地，就是存放 x 和 y 的地方。

### 4. 为什么要这样设计？（设计原理）

1.  **自动回收：** 当函数执行结束时，只需要执行**函数尾声 (Function Epilogue)**，把 SP 加回去（add sp, sp, \#8），这块内存就自动“销毁”了。这就是为什么局部变量不需要手动 free 的原因。

2.  **支持递归：** 每次调用函数，SP 都会再减一次，形成一个新的独立空间。这保证了递归函数每一层都有自己独立的局部变量，互不干扰。

3.  **确定性：** 偏移量在**编译时**就计算好了。CPU 只需要一条减法指令就能瞬间完成内存分配，效率极高。

### 💡 嵌入式避坑指南

如果你在 Zynq 或 RK3568 上写驱动，定义了一个巨大的局部数组（例如 char buf\[4096\]），编译器会生成 sub sp, sp, \#4096。

如果此时内核或任务的栈空间一共只有 8KB，而你已经嵌套调用了好几层，这个巨大的“偏移量”可能会让 SP 直接**冲出栈底**，覆盖掉底部的进程描述符或其它关键数据，导致严重的 Stack Overflow。
