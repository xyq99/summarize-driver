在计算机内存管理中，**栈（Stack）和堆（Heap）的增长方向通常是相对而行**的。这种设计是为了最大限度地利用地址空间，防止两者过早发生碰撞。

以下是它们具体的增长方向及逻辑：

### 1. 栈空间（Stack）：向低地址增长

在大多数现代操作系统（如 Linux、Windows）和主流架构（如 x86/x64）中，栈是**从高地址向低地址**增长的。

- **初始状态：** 栈底位于内存的高地址端。

- **入栈操作（Push）：** 当程序调用函数或分配局部变量时，栈指针（Stack Pointer, ESP/RSP）会向**减小**的方向移动。

- **物理隐喻：** 想象一个倒挂的桶，你往里面放东西，液面反而离地面越近（地址数值变小）。

### 2. 堆空间（Heap）：向高地址增长

堆空间通常由程序员手动分配（如 C 语言中的 malloc 或 Java 中的 new），它是**从低地址向高地址**增长的。

- **初始状态：** 堆起始于进程数据段（Data Segment）之后的低地址区域。

- **分配操作：** 当申请更多内存时，堆指针向**增加**的方向移动。

- **管理方式：** 堆的内存是不连续的，由链表或更复杂的算法管理，但总体扩张趋势是向上的。

### 内存布局示意图

为了更直观地理解，可以参考下表：

| 特性         | 栈 (Stack)                    | 堆 (Heap)                     |
|--------------|-------------------------------|-------------------------------|
| **增长方向** | **向下**（高地址 \to 低地址） | **向上**（低地址 \to 高地址） |
| **分配方式** | 自动分配（系统管理）          | 手动分配（程序员管理）        |
| **管理结构** | 连续的线性空间                | 链表或树形结构（非连续）      |
| **速度**     | 极快（仅移动指针）            | 较慢（涉及内存查找与合并）    |

### 为什么这样设计？

这种“两头对跑”的设计（堆向上，栈向下）主要是为了**内存利用效率**。

**核心逻辑：** 既然我们无法预知一个程序是需要更多的堆还是更多的栈，不如让它们分别从虚拟地址空间的两端开始生长。这样，中间剩下的空闲内存就可以根据需要动态地分配给其中任何一方，直到两者“相遇”引发溢出（Stack Overflow 或 Heap Overflow）。

如果你是在面试中遇到这个问题，记住这个口诀：**“栈下堆上”**。
