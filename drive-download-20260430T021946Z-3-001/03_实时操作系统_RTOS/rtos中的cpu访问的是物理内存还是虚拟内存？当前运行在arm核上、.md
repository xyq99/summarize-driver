这是一个非常经典的问题。在嵌入式实时操作系统（RTOS）的语境下，答案通常是：**CPU 访问的是物理内存（Physical Memory）。**

不过，为了严谨起见，我们需要根据 ARM 处理器的硬件特性和 RTOS 的设计哲学来详细拆解。

## 1. 为什么 RTOS 倾向于物理内存？

与 Linux 或 Windows 等通用操作系统（GPOS）不同，RTOS 的核心目标是**确定性（Determinism）**。

- **消除延迟：** 虚拟内存通常依赖于 **MMU（内存管理单元）** 和页表查询。如果发生“缺页异常（Page Fault）”，CPU 必须去硬盘或 Flash 中交换数据，这会产生巨大的、不可预测的延迟。

- **简化开销：** RTOS 运行的环境（如 ARM Cortex-M 系列）通常资源有限。维护复杂的虚拟地址映射表会消耗额外的 RAM 和 CPU 周期。

- **直接映射：** 在大多数 ARM RTOS（如 FreeRTOS, uC/OS, ThreadX）中，程序中的地址 0x20000000 对应的就是芯片内部 SRAM 的 0x20000000。

## 2. ARM 核的不同表现

ARM 的架构决定了它如何处理内存。我们需要区分你具体使用的是哪种“核”：

### ARM Cortex-M 系列 (M0, M3, M4, M7, M33 等)

这是 RTOS 最常用的内核。

- **无 MMU：** 这些内核根本没有内存管理单元（MMU），因此**无法实现**虚拟地址映射。

- **MPU（内存保护单元）：** 虽然没有虚拟内存，但它们通常配有 MPU。MPU 不改变地址（输入 = 输出），它只负责检查当前任务是否有权访问该物理地址。

### ARM Cortex-A 系列 (A7, A53, A72 等)

这是性能级内核，通常运行 Linux，但也可以运行 RTOS（如 QNX 或运行在特定的实时内核上）。

- **支持虚拟内存：** 这些内核拥有完整的 MMU。

- **RTOS 的选择：**

  - 即使在 Cortex-A 上，许多 RTOS 也会配置 MMU 为 **“对等映射”（Identity Mapping）**。也就是说，虚拟地址等于物理地址，目的是利用 MMU 的内存权限检查功能，而不使用其地址转换功能。

  - **例外：** 像 **QNX** 这种微内核 RTOS 确实会使用完整的虚拟内存机制，以实现进程间的深度隔离。

## 3. 对比总结

| **特性**     | **通用操作系统 (Linux/Win)** | **典型 RTOS (FreeRTOS/RT-Thread)** |
|--------------|------------------------------|------------------------------------|
| **内存模式** | 虚拟内存 (Virtual Memory)    | **物理内存 (Physical Memory)**     |
| **地址映射** | 复杂映射 (VA \$\neq\$ PA)    | 1:1 映射或直接访问                 |
| **实时性**   | 较低（存在缺页延迟）         | **极高（执行时间可预测）**         |
| **硬件依赖** | 必须有 MMU                   | 仅需 CPU，可选 MPU                 |

## 结论

在你的 ARM 核上运行 RTOS 时，你编写的代码中所操作的指针地址，**绝大多数情况下就是实际的物理硬件地址**。

**特殊情况提醒：**

如果你使用的是高性能 ARM 内核（Cortex-A）并且运行的是支持 POSIX 标准的复杂 RTOS，它可能启用了虚拟地址。但对于 90% 的 ARM 嵌入式开发场景，你接触到的都是物理内存。
