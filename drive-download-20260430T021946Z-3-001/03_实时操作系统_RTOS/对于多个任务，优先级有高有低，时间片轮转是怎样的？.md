在 RTOS 中，**时间片轮转（Time Slicing）** 并不是对所有任务进行“大锅饭”式的轮询。它的执行有一个铁律：**只有当前处于就绪态的最高优先级任务，才能参与时间片的分配。**

我们可以把这种机制拆解为两个核心逻辑：**“优先级决定生死”** 和 **“时间片决定顺序”**。

### 1. 核心逻辑：优先级绝对优先

RTOS 的内核始终在执行一个动作：**寻找当前所有就绪任务中优先级最高的那一个，并把它推上 CPU。**

- **如果高优先级任务存在：** 只要高优先级任务不进入阻塞（如调用 ）或挂起状态，低优先级的任务**永远没有机会**运行，即使它们已经排队等了很久。这种情况被称为“任务饥饿”。vTaskDelay

- **如果存在多个同级任务：** 当最高优先级这一层里不止一个任务时，**时间片轮转**才真正开始介入。

### 2. 场景演示：时间片如何运作

假设我们有三个任务，优先级如下：

- **任务 A**：优先级 3（高）

- **任务 B**：优先级 2（中）

- **任务 C**：优先级 2（中）

#### 阶段一：高优先级任务独占

当任务 A 处于就绪态时，CPU 只会跑任务 A。此时任务 B 和 C 都在后面等着，没有任何时间片的概念。

#### 阶段二：同级任务轮转

如果任务 A 调用了 进入阻塞态，或者被挂起了。现在最高优先级变成了 **2**，而这个等级下有两个任务（B 和 C）。vTaskDelay

1.  **Tick 1：** 任务 B 运行。

2.  **Tick 中断触发：** 内核检查发现同级（优先级 2）还有另一个就绪任务 C。

3.  **切换：** 内核触发 PendSV，把任务 B 换下来，任务 C 换上去。

4.  **Tick 2：** 任务 C 运行。

5.  **循环：**B -\> C -\> B -\> C，以此类推。

#### 阶段三：高优先级抢占

在任务 B 运行的中途，如果任务 A 的延时时间到了，或者被某个中断唤醒了。

- **即刻抢占：** 内核会立即打断任务 B（或者正在轮转的任务 C），直接把 CPU 给 A。任务 B 和 C 的时间片轮转立即暂停，重新回到等待状态。

### 3. 总结对照表

| **场景**               | **调度行为**   | **备注**                                             |
|------------------------|----------------|------------------------------------------------------|
| **优先级不同**         | **抢占式调度** | 高优先级任务只要就绪，立即执行，不谈时间片。         |
| **优先级相同**         | **时间片轮转** | 只有当同级任务是当前最高级时，才互相换班。           |
| **只有一个最高级任务** | **独占执行**   | 即使开启了时间片，它也会一直运行（因为没同级可换）。 |

### 4. 关键配置提示

在 FreeRTOS 中，这个行为由两个宏共同控制：

- configUSE_PREEMPTION(抢占)：必须为 ，否则任务不会主动让位。1

- configUSE_TIME_SLICING(时间片)：

  - 设为 ：同级任务在每个 Tick 中断时强制换班。1

  - 设为 ：同级任务不会主动换班，除非当前任务主动阻塞或调用了 。0taskYIELD()

**设计建议：**

在嵌入式开发中，通常建议给不同的功能模块设置不同的优先级。**时间片轮转只是一种保底机制**，真正的实时性应该依靠合理的优先级划分来实现。
