RTOS（实时操作系统）中的队列通常支持**阻塞**和**非阻塞**两种操作模式。具体使用哪种方式，取决于调用 API 时设置的**超时参数（Timeout）**。

### 1. 非阻塞模式 (Non-blocking)

当 API 的超时参数设置为 **0** 时，操作为非阻塞。

- **发送数据时：** 如果队列已满，函数会立即返回一个错误码（如 errQUEUE_FULL），而不会让当前任务进入等待状态。

- **接收数据时：** 如果队列为空，函数会立即返回（如 errQUEUE_EMPTY），任务继续执行后续代码。

- **适用场景：** 适用于中断服务函数（ISR）或对实时性要求极高、不允许任何延迟的任务。

### 2. 阻塞模式 (Blocking)

当超时参数设置为 **大于 0** 或 **无限等待（Include / PortMAX_DELAY）** 时，操作为阻塞。

- **发送阻塞：** 如果队列已满，任务会被挂起并进入“阻塞态”。一旦队列有了空位，操作系统会唤醒该任务。如果超过了设定的时间队列仍满，则任务醒来并返回超时错误。

- **接收阻塞：** 如果队列为空，任务会挂起。直到其他任务向队列发送了数据，操作系统才会唤醒该任务进行处理。

- **适用场景：** 常用于任务间的同步与通信，能够有效降低 CPU 占用率，因为阻塞的任务不占用执行时间。

### 队列状态与任务切换逻辑

| 场景               | 操作类型 | 队列状态 | 行为结果                     |
|--------------------|----------|----------|------------------------------|
| **入队 (Send)**    | 阻塞     | 已满     | 任务进入阻塞态，等待空间释放 |
| **入队 (Send)**    | 非阻塞   | 已满     | 立即返回失败代码             |
| **出队 (Receive)** | 阻塞     | 为空     | 任务进入阻塞态，等待新数据   |
| **出队 (Receive)** | 非阻塞   | 为空     | 立即返回失败代码             |

### 核心机制：等待列表 (Waiting List)

在 FreeRTOS 或 uC/OS 等内核中，队列结构体内部维护了两个等待列表：

1.  **Tasks Waiting to Send：** 存放因队列满而阻塞的发送任务。

2.  **Tasks Waiting to Receive：** 存放因队列空而阻塞的接收任务。

当队列状态改变（如从满变为不满）时，内核会根据**优先级**从对应的等待列表中唤醒任务。如果优先级相同，则通常遵循 **FIFO**（先来先服务）原则。
